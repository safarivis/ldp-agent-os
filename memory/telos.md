# telos

## Purpose

This file stores the strategic direction and long-term aim of the `_agent` system.

## Current direction

Build a user-global, portable agent operating system that:
- keeps memory in markdown
- routes work through clear contexts
- captures expert taste as a compounding asset
- stays simple enough to maintain
- can later be replicated and compared across users or test setups
- helps Louis and RevvTech run many client/project repos through a shared operating layer that compounds learnings over time

## North star

Create a practical delivery operating system for Louis and RevvTech where:
- each client/project has its own repo as project-specific truth
- `_agent` holds the reusable cross-project standards, workflows, templates, and learnings
- Git/GitHub provide shared history, versioning, and collaboration
- pi/agents help execute work inside structured repos, not only chats
- human decisions still flow through email, messaging, and meetings, then get written back into the system
- repeatable patterns are noticed, captured, and promoted into reusable assets

Success looks like:
- faster setup for each new project
- better handover and continuity
- less re-solving of the same operational problems
- clearer separation between local project truth and reusable system truth
- compounding leverage across multiple RevvTech projects

## Long-term goals

- Establish `_agent` as the canonical agent control plane for `ldp`
- Reduce workflow chaos through structured routing and review
- Turn corrections, decisions, and standards into reusable assets
- Support future comparison with `/home/test/_agent`
- Eventually add stewardship/critic agents only after the manual workflow is proven

## Strategic preference order

When tradeoffs appear, prefer:
1. improving the live canonical system over speculative side systems
2. reusable templates and workflows over one-off cleverness
3. validated iteration over broad redesign
4. portability and clarity over local hacks
