# Info Needed From Dom

## Recommendation

**Do light structure first, then get Dom to fill in the real operating details.**

This is the best order.

### Why
Because:
- a blank page is hard to react to
- a simple draft gives Dom something concrete to confirm or correct
- you avoid waiting for perfect info before getting organized

So the answer is:

## Best order
1. create the simple structure first
2. keep it lightweight and clearly marked as draft
3. ask Dom to confirm the real operating details
4. then fill in names, ownership, permissions, and lifecycle rules

---

## What to ask Dom first

### Team / roles
- who are the main people and what do they own?
- who approves strategy, delivery, architecture, and client communication?
- who is client-facing per account?
- who reviews final quality?

### Client separation
- which clients can live in the shared company AgentOS?
- which clients need stricter isolation?
- what should partners be allowed to access?

### Project lifecycle
- what are RevvTech’s real project stages?
- when does a lead become an active project?
- who approves handover / completion?

### Partner model
- how are network partners brought in?
- who owns them internally?
- should they work in GitHub only or also inside `_agent`?

## Suggested message to Dom

Hi Dom,

I’ve put together a lightweight draft operating structure for how RevvTech could use AgentOS across the team, projects, and partners.

Before we hard-code anything, I want to confirm a few basics with you so we don’t build the wrong model.

The main things to confirm are:
- team roles and approvals
- client separation / access rules
- project lifecycle stages
- how network partners should fit in

I’ve created draft structures first so we have something concrete to react to, rather than starting from a blank page.

Once you confirm the basics, we can fill in the real ownership and operating rules properly.

---

## KISS advice

Do **not** wait for all info before creating structure.

Also do **not** fully finalize the structure before Dom confirms the operating model.

The right move is:
- draft first
- confirm second
- refine third
