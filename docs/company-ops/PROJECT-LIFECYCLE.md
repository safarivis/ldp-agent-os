# Project Lifecycle

## Recommendation

**Create the lifecycle structure first, then confirm the exact stages with Dom.**

This gives agents and humans a shared operating model early.

---

## Starter lifecycle

### 1. Lead / Opportunity
Work here:
- notes
- discovery questions
- fit assessment
- early client context

### 2. Scope / Discovery
Work here:
- requirements
- constraints
- initial architecture
- estimates
- risks

### 3. Proposal / Approval
Work here:
- proposal drafts
- commercial notes
- decision points
- approval record

### 4. Active Delivery
Work here:
- GitHub issues
- implementation
- updates
- review notes
- blockers

### 5. Review / Handover
Work here:
- QA
- docs
- client handover items
- sign-off

### 6. Support / Improvement
Work here:
- bug fixes
- follow-up tasks
- optimizations
- retained support context

### 7. Archive
Work here:
- final notes
- final decisions
- retained useful context

---

## KISS rule

Do not over-model the lifecycle.

Use 5–7 simple stages only.

The point is not bureaucracy.
The point is to make status and handoffs clear.

---

## Default operating rule

- GitHub tracks execution during active delivery
- `_agent` tracks context, notes, and reusable knowledge across the lifecycle
- humans approve stage changes that matter commercially or operationally
