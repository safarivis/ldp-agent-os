# Partner Working Rules

## Recommendation

**Create the rule structure first, then confirm the real partner model with Dom.**

Because RevvTech uses network partners, this matters early.

---

## Why this matters

Partners can add delivery capacity, but they also increase risk if access and ownership are not clear.

---

## Starter rules

### 1. Least access by default
Partners should only see:
- the project they are working on
- the files and repos they need
- the issues and context needed for delivery

### 2. No cross-client visibility by default
A partner working on Client A should not see Client B context unless explicitly approved.

### 3. GitHub first for delivery work
If possible, partners should work mainly through:
- GitHub issues
- pull requests
- clearly scoped docs

### 4. `_agent` access only when needed
If a partner needs `_agent` access, scope it tightly.

### 5. Human owner always exists
Every partner contribution should have:
- one RevvTech owner
- one review path
- one approval path

---

## Questions for Dom

1. How do partners currently work with RevvTech?
2. Do partners need direct repo access?
3. Do partners need `_agent` access, or should they stay in GitHub only?
4. Who owns partner review and sign-off?
5. What client approval is needed before a partner is involved?

---

## Default rule until confirmed

- partners do not get root `_agent` access by default
- partners work in project-specific repos/issues where possible
- partner work is reviewed by a RevvTech human owner
- sensitive client context stays restricted
